# Провизионинг устройства — Архитектура

Дизайн-документ: как покупатель настраивает модуль VehicleLogger и привязывает к своему аккаунту.

---

## Проблема

Сейчас WiFi, Device ID и API-ключ захардкожены в `config.h`. Для продакшена нужно:

1. **WiFi** — каждый покупатель использует свою сеть/хотспот
2. **Привязка к аккаунту** — модуль должен принадлежать конкретной компании
3. **Изоляция** — компания A не видит устройства компании B (мультитенантность)
4. **Безопасность** — API-ключ не должен быть захардкожен в прошивке

---

## Жизненный цикл устройства

```
  ЗАВОД             ПОКУПАТЕЛЬ              ЭКСПЛУАТАЦИЯ
    |                   |                       |
    v                   v                       v
[Изготовлено]  →  [Не настроено]  →  [Привязано / Работает]
                       |                       |
                  (SoftAP режим)          [Смена WiFi] → обратно в "Не настроено"
                  Настройка WiFi               |
                  Привязка к аккаунту     [Отвязка] → обратно в "Изготовлено"
                       |
                       v
                  [Привязано / Работает]
```

### Состояния

| Состояние | WiFi в NVS | API-ключ в NVS | Что делает устройство |
|-----------|------------|-----------------|----------------------|
| Изготовлено | — | — | SoftAP (captive portal) |
| WiFi настроен | Да | — | Подключено к WiFi, поллит `/provision` |
| Привязано | Да | Да | Отправляет телеметрию (рабочий режим) |

## 1. Заводская подготовка

Прошивка **одинакова** для всех устройств. Отличаются только данные в NVS.

### Что записывается в NVS при производстве

| NVS Namespace | Ключ | Значение | Описание |
|---------------|------|----------|----------|
| `factory` | `serial` | `VL-A3F82B01` | Серийный номер (уникальный) |
| `factory` | `secret` | 32 случайных байта | Секрет устройства |

### Серийный номер

Формат: `VL-XXXXXXXX` — 8 hex-символов, вычисляется из MAC-адреса ESP32.

```
MAC: AA:BB:CC:DD:EE:FF → Serial: VL-CCDDEEFF
```

Каждый ESP32 имеет уникальный MAC — серийники не пересекутся.

### Секрет устройства

32 случайных байта (256 бит), генерируются при производстве. Хранятся только на устройстве (в NVS) и на бэкенде (в виде хеша).

### QR-код на корпусе

Наклейка с QR-кодом содержит:

```
https://app.vehiclelogger.ru/claim?sn=VL-A3F82B01&ds=A1B2C3D4
```

- `sn` — серийный номер (полный)
- `ds` — первые 8 hex-символов секрета (фрагмент, не весь секрет)

Фрагмент секрета нужен для верификации: пользователь, сканирующий QR, физически рядом с устройством.

### Заводской скрипт (Python + esptool)

Запускается один раз на каждое устройство через USB:

```
1. Прошить firmware.bin (одинаковый для всех)
2. Прочитать MAC-адрес ESP32
3. Сгенерировать серийный номер из MAC
4. Сгенерировать 32 случайных байта (секрет)
5. Записать serial + secret в NVS-партицию
6. Вывести данные для печати QR-наклейки
7. Отправить serial + hash(secret) в БД бэкенда (batch API или CSV)
```

## 2. Настройка WiFi (SoftAP Captive Portal)

### Сценарий пользователя

1. Покупатель устанавливает модуль в фуру (OBD-II)
2. Модуль включается. WiFi-кредов нет → запускается **SoftAP**
3. На телефоне в списке WiFi появляется `VehicleLogger-CCDDEEFF`
4. Подключается к этой сети (пароль: `setup1234`, указан на наклейке)
5. Открывается **captive portal** — веб-страница настройки
6. Выбирает свою WiFi-сеть из списка, вводит пароль, нажимает "Подключить"
7. Модуль подключается к WiFi → сохраняет креды в NVS

### Что делает ESP32

```
BOOT
  |
  Читаем NVS: wifi/ssid
  |
  ├── Нет кредов → Режим SoftAP
  │     1. Сканируем WiFi-сети (получаем список SSID + RSSI)
  │     2. Запускаем AP: "VehicleLogger-{serial_last8}", пароль "setup1234"
  │     3. Запускаем DNS-сервер (все запросы → на себя, captive portal)
  │     4. Запускаем HTTP-сервер на порту 80
  │     5. Раздаём страницу настройки (HTML из PROGMEM или LittleFS)
  │     6. Ждём POST с SSID + паролем
  │     7. Пробуем подключиться к указанной сети
  │     8. Если OK → сохраняем в NVS, выключаем AP, переходим дальше
  │     9. Если FAIL → отвечаем ошибкой, пользователь может повторить
  │
  └── Есть креды → Пробуем подключиться
        ├── OK → Переходим к проверке API-ключа (раздел 3)
        └── FAIL (5 попыток) → Режим SoftAP (креды устарели)
```

### Captive Portal (веб-страница)

Минимальная HTML-страница (~5 КБ), хранится в прошивке:

```
┌────────────────────────────────┐
│   VehicleLogger — Настройка    │
│                                │
│   Выберите WiFi-сеть:          │
│   ┌──────────────────────┐     │
│   │ ▼ Keenetic-5935  -45│     │
│   │   TP-Link_Guest  -72│     │
│   │   MTS-Router     -80│     │
│   └──────────────────────┘     │
│                                │
│   Пароль: [____________]       │
│                                │
│   [ Подключить ]               │
│                                │
│   Серийный номер: VL-A3F82B01  │
│   Статус: ожидание настройки   │
└────────────────────────────────┘
```

### Локальные HTTP-эндпоинты (на ESP32)

| Путь | Метод | Описание |
|------|-------|----------|
| `/` | GET | Главная страница (HTML) |
| `/api/scan` | GET | Список WiFi-сетей: `[{"ssid":"...","rssi":-45,"secure":true}]` |
| `/api/connect` | POST | Подключение: `{"ssid":"...","password":"..."}` |
| `/api/status` | GET | Текущий статус: `{"wifi":"connected","ip":"...","claimed":false}` |

## 3. Привязка к аккаунту (Claim)

После настройки WiFi устройство подключено к сети, но ещё не привязано к аккаунту и не имеет API-ключа.

### Сценарий пользователя

1. Captive portal показывает: "WiFi подключён! Теперь отсканируйте QR-код на устройстве для привязки к аккаунту."
2. Пользователь открывает камеру телефона → сканирует QR-код на корпусе
3. Открывается `https://app.vehiclelogger.ru/claim?sn=VL-A3F82B01&ds=A1B2C3D4`
4. Если не авторизован → редирект на логин → возврат на claim-страницу
5. Страница показывает: "Привязать устройство VL-A3F82B01?"
6. Пользователь выбирает фуру из списка (или создаёт новую) → нажимает "Привязать"
7. Бэкенд генерирует API-ключ → привязывает устройство к тенанту

### Что делает ESP32 (параллельно)

```
WiFi подключён, API-ключа нет
  |
  Цикл: каждые 10 сек
  |
  POST /api/devices/provision
    Body: { "serialNumber": "VL-A3F82B01", "deviceSecret": "полный_секрет_hex" }
  |
  ├── 202 Accepted → ещё не привязано, повторяем через 10 сек
  ├── 200 OK → получен API-ключ!
  │     { "apiKey": "xxxxxxxx", "backendUrl": "https://...", "sendIntervalMs": 5000 }
  │     Сохраняем в NVS → переходим в рабочий режим
  └── 403 Forbidden → секрет не совпадает (ошибка — мигаем LED)
```

### Sequence Diagram

```
  Телефон              ESP32                     Бэкенд
     |                   |                          |
     |  [SoftAP: WiFi настроен]                     |
     |                   |                          |
     |                   |--- POST /provision ----->|
     |                   |<-- 202 (ждём claim) -----|
     |                   |                          |
     | -- Сканирует QR --|------------------------->|
     |    (в браузере)   |    POST /devices/claim   |
     |                   |    + JWT + sn + ds        |
     |<-- "Привязано!" --|--------------------------|
     |                   |                          |
     |                   |--- POST /provision ----->|
     |                   |<-- 200 + API-ключ -------|
     |                   |                          |
     |                   |--- POST /telemetry ----->|
     |                   |     (рабочий режим)      |
```

## 4. Мультитенантность

### Концепция

Каждая компания-покупатель = **тенант** (tenant). Все данные изолированы: компания видит только свои устройства, фуры, водителей.

### Изменения в БД

#### Новая таблица: `tenants`

```sql
CREATE TABLE tenants (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(200) NOT NULL,   -- "ООО Транспорт Логистик"
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### Изменения в существующих таблицах

```sql
-- devices: добавить tenant_id, расширить статусы
ALTER TABLE devices ADD COLUMN tenant_id       UUID REFERENCES tenants(id);
ALTER TABLE devices ADD COLUMN device_secret_hash VARCHAR(128) NOT NULL;
ALTER TABLE devices ADD COLUMN status          VARCHAR(20) NOT NULL DEFAULT 'manufactured';
        -- manufactured | claimed | active | deactivated
ALTER TABLE devices ADD COLUMN claimed_at      TIMESTAMPTZ;
ALTER TABLE devices ADD COLUMN claimed_by      UUID REFERENCES users(id);

-- vehicles: привязать к тенанту
ALTER TABLE vehicles ADD COLUMN tenant_id UUID NOT NULL REFERENCES tenants(id);

-- users: привязать к тенанту
ALTER TABLE users ADD COLUMN tenant_id UUID NOT NULL REFERENCES tenants(id);
```

#### Новая таблица: `provisioning_tokens`

```sql
CREATE TABLE provisioning_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES devices(id),
    api_key     VARCHAR(128) NOT NULL,       -- plaintext, короткоживущий
    expires_at  TIMESTAMPTZ NOT NULL,        -- +10 минут
    consumed    BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Как работает изоляция

```
Компания A (tenant_id = AAA)          Компания B (tenant_id = BBB)
├── Фура "МАН А123" (vehicle)         ├── Фура "Скания Б456" (vehicle)
├── Устройство VL-001 (device)        ├── Устройство VL-005 (device)
├── Устройство VL-002 (device)        ├── Устройство VL-006 (device)
├── Админ Иван (user, admin)          ├── Админ Пётр (user, admin)
└── Водитель Олег (user, driver)      └── Водитель Сергей (user, driver)

Запросы фильтруются: WHERE tenant_id = current_user.tenant_id
→ Иван видит только VL-001, VL-002 и "МАН А123"
→ Пётр видит только VL-005, VL-006 и "Скания Б456"
```

## 5. API-эндпоинты провизионинга

### Со стороны устройства (вызывает ESP32)

#### `POST /api/devices/provision`

ESP32 вызывает этот эндпоинт для получения API-ключа после настройки WiFi.

**Запрос:**
```json
{
  "serialNumber": "VL-A3F82B01",
  "deviceSecret": "a1b2c3d4e5f6...полный_секрет_64_hex_символа"
}
```

**Ответы:**

| Код | Ситуация | Тело ответа |
|-----|----------|-------------|
| 200 | Устройство привязано, ключ готов | `{"apiKey":"...","backendUrl":"https://...","sendIntervalMs":5000}` |
| 202 | Ещё не привязано, ждём claim | `{"status":"waiting_for_claim"}` |
| 403 | Неверный секрет | `{"error":"invalid_secret"}` |
| 410 | Устройство деактивировано | `{"error":"deactivated"}` |

**Безопасность:** Только по HTTPS. Полный секрет устройства подтверждает физическое владение.

### Со стороны веб-панели (вызывает пользователь)

Все требуют JWT-авторизации.

#### `POST /api/devices/claim`

Привязка устройства к аккаунту (после сканирования QR).

**Запрос:**
```json
{
  "serialNumber": "VL-A3F82B01",
  "deviceSecretFragment": "A1B2C3D4",
  "vehicleId": "uuid-фуры-или-null"
}
```

**Ответы:**

| Код | Ситуация |
|-----|----------|
| 200 | Успех: `{"deviceId":"uuid","serialNumber":"VL-A3F82B01","status":"claimed"}` |
| 404 | Серийный номер не найден |
| 400 | Фрагмент секрета не совпадает |
| 409 | Устройство уже привязано к другому тенанту |

**Логика бэкенда:**
1. Найти устройство по серийному номеру
2. Проверить фрагмент секрета
3. Проверить что устройство не привязано (status = `manufactured`)
4. Сгенерировать API-ключ (64 hex)
5. Обновить: `tenant_id`, `vehicle_id`, `status = claimed`, `api_key_hash = hash(key)`
6. Создать provisioning_token (api_key plaintext, expires +10 мин)

#### `POST /api/devices/{id}/unclaim`

Отвязка устройства (передача, продажа).

- Очищает `tenant_id`, `vehicle_id`, `api_key_hash`
- Статус → `manufactured`
- Устройство при следующем запросе получит 410 → сбросит ключ → начнёт поллинг заново

#### `PUT /api/devices/{id}/vehicle`

Переназначить устройство на другую фуру (внутри того же тенанта).

```json
{ "vehicleId": "новый-uuid" }
```

#### `POST /api/devices/{id}/rotate-key`

Ротация API-ключа (при компрометации).

- Генерирует новый ключ, обновляет хеш
- Устройство получит 401 → сбросит ключ из NVS → перейдёт в режим поллинга → получит новый ключ

#### `GET /api/devices`

Список устройств текущего тенанта.

```json
[
  {
    "id": "uuid",
    "serialNumber": "VL-A3F82B01",
    "vehicleName": "МАН А123БВ",
    "status": "active",
    "isOnline": true,
    "lastPingAt": "2026-03-29T10:00:00Z"
  }
]
```

## 6. Изменения прошивки (план реализации)

### NVS вместо config.h

| Сейчас (config.h) | Будет (NVS) |
|-------|------|
| `WIFI_SSID` | `wifi/ssid` |
| `WIFI_PASSWORD` | `wifi/password` |
| `DEVICE_ID` | `factory/serial` (записано на заводе) |
| `DEVICE_API_KEY` | `cloud/api_key` (получено при claim) |
| `BACKEND_URL` | `cloud/backend_url` (получено при claim) |

В `config.h` остаются только **пины** и **аппаратные константы**:

```c
#pragma once

// ===== CAN (TWAI) =====
#define CAN_TX_PIN    GPIO_NUM_22
#define CAN_RX_PIN    GPIO_NUM_21

// ===== GPS (NEO-M8N) =====
#define GPS_RX_PIN    16
#define GPS_TX_PIN    17
#define GPS_BAUD      9600

// ===== IMU (MPU-6050) =====
#define MPU_SDA_PIN   18
#define MPU_SCL_PIN   19
#define MPU_ADDR      0x68

// ===== Провизионинг =====
#define AP_PASSWORD           "setup1234"
#define RESET_BUTTON_PIN      GPIO_NUM_0    // кнопка BOOT на DevKit
#define WIFI_RETRY_COUNT      5
#define PROVISION_POLL_MS     10000         // интервал поллинга
#define DEFAULT_BACKEND_URL   "https://api.vehiclelogger.ru"

// ===== LED =====
#define LED_PIN               GPIO_NUM_2    // встроенный LED
```

### Boot State Machine

```
                          BOOT
                           |
                    Читаем NVS wifi/ssid
                           |
                  +--------+--------+
                  |                 |
             Нет кредов       Есть креды
                  |                 |
           SoftAP режим      Пробуем WiFi
           (captive portal)        |
                            +------+------+
                            |             |
                       Подключено    Не удалось (5 раз)
                            |             |
                     Читаем NVS      SoftAP режим
                     cloud/api_key
                            |
                   +--------+--------+
                   |                 |
              Нет ключа        Есть ключ
                   |                 |
            Поллинг /provision  РАБОЧИЙ РЕЖИМ
                   |           (телеметрия + heartbeat)
              Получили ключ
                   |
            Сохранили в NVS
                   |
              РАБОЧИЙ РЕЖИМ
```

### Структура файлов прошивки (целевая)

```
firmware/src/
├── main.cpp              — setup() + loop(), state machine
├── provisioning.h/.cpp   — SoftAP, DNS, HTTP-сервер, captive portal
├── wifi_manager.h/.cpp   — подключение к WiFi, реконнект, NVS-креды
├── cloud.h/.cpp          — HTTP: provision, telemetry, heartbeat
├── nvs_store.h/.cpp      — обёртка над NVS: read/write factory, wifi, cloud
├── can_reader.h/.cpp     — CAN/TWAI (этап 2)
├── gps_reader.h/.cpp     — GPS NMEA (этап 3)
├── imu_reader.h/.cpp     — MPU-6050 (этап 4)
└── led_status.h/.cpp     — индикация состояния через LED
```

### Кнопка сброса

Используем GPIO 0 (кнопка BOOT на DevKit). В продакшен-плате — отдельная кнопка.

| Действие | Удержание | Что сбрасывается | Что сохраняется |
|----------|-----------|-------------------|-----------------|
| Сброс WiFi | 5 секунд | `wifi/ssid`, `wifi/password` | serial, secret, api_key |
| Factory reset | 10 секунд | Всё (wifi + cloud) | serial, secret (заводские) |

После сброса WiFi → устройство входит в SoftAP, можно ввести новые креды. API-ключ сохраняется — не нужно заново привязывать.

После factory reset → устройство как новое. Нужна повторная привязка через QR.

### LED-индикация

| Паттерн | Состояние |
|---------|-----------|
| Медленное мигание (1 Гц) | SoftAP режим — ожидание настройки WiFi |
| Быстрое мигание (4 Гц) | Подключение к WiFi / поллинг provision |
| Горит постоянно | Рабочий режим, всё ОК |
| Двойная вспышка каждые 3 сек | Ошибка (нет WiFi после настройки) |

## 7. Безопасность

### Транспорт

- **ESP32 → Бэкенд**: только HTTPS. Прошивка содержит корневой CA-сертификат (Let's Encrypt).
- **SoftAP**: локальная сеть, пароль на наклейке. WiFi-пароль передаётся по HTTP (локально, не в интернет). При необходимости — self-signed TLS на ESP32.

### API-ключи

- Генерируются на бэкенде при claim (64 hex = 256 бит)
- Бэкенд хранит только **хеш** (SHA-256)
- Plaintext существует только: в provisioning_token (10 мин) и в NVS устройства
- При компрометации — ротация через `/rotate-key`

### Защита claim

Для привязки устройства нужны **оба фактора**:
1. JWT авторизованного пользователя (знание — логин/пароль)
2. Фрагмент секрета с QR-кода (физический доступ к устройству)

Злоумышленник, знающий только серийный номер, не может привязать устройство — нужен физический доступ к наклейке.

### Защита provision

- Полный секрет (256 бит) подтверждает что вызов идёт с реального устройства
- QR содержит только фрагмент (32 бита) — недостаточно для вызова `/provision`
- Запрос идёт по HTTPS → секрет не перехватывается

### Ротация и отзыв

| Сценарий | Действие |
|----------|----------|
| Подозрение на компрометацию ключа | `/rotate-key` → устройство получит новый ключ автоматически |
| Продажа/передача устройства | `/unclaim` → устройство теряет доступ, новый владелец привязывает заново |
| Потерянное устройство | `/unclaim` + деактивация на бэкенде |

## 8. Смена WiFi-сети

Когда устройство переезжает в другую фуру с другим хотспотом:

1. Удерживать кнопку сброса **5 секунд** → LED быстро мигнёт 3 раза
2. Устройство перезагружается в SoftAP-режим
3. Подключиться телефоном, ввести новые WiFi-креды
4. Устройство подключается к новой сети с **тем же API-ключом** — не нужно заново привязывать
5. Если нужно переназначить на другую фуру → через веб-панель (`PUT /devices/{id}/vehicle`)

| `ESPAsyncWebServer` | HTTP-сервер для captive portal |
| `AsyncTCP` | Async TCP для ESPAsyncWebServer |
| `DNSServer` (встроенная) | Перенаправление DNS для captive portal |
| `Preferences` (встроенная) | NVS read/write |
| `WiFiClientSecure` (встроенная) | HTTPS для связи с бэкендом |

### Бэкенд (Тимур)

- Таблица `tenants` + `tenant_id` во всех таблицах
- Эндпоинты: `/devices/provision`, `/devices/claim`, `/devices/{id}/unclaim`, `/devices/{id}/vehicle`, `/devices/{id}/rotate-key`
- JWT содержит `tenant_id` → middleware фильтрует все запросы
- Страница claim на фронтенде (Vue 3)
