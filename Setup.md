# Setup — Первый этап проекта

## Цель этапа

Поднять скелет всей системы: каждый компонент запускается, компоненты связаны между собой, данные проходят сквозь всю цепочку от устройства до интерфейсов. Без финальной логики и красоты — только рабочий каркас.

---

## Результат этапа

```
ESP32 (тестовые данные) → Бэкенд → PostgreSQL → Vue 3 (таблица с данными)
                                        ↓
                                   Telegram-бот (эхо-ответ на команды)
```

Администратор открывает веб-панель и видит тестовые данные от устройства. Водитель пишет боту `/status` и получает ответ. Всё соединено.

---

## Сергей — Устройство (ESP32)

### Задачи

1. **Настроить среду разработки**
   - Установить ESP-IDF или PlatformIO
   - Создать проект прошивки в репозитории (`/firmware`)
   - Убедиться, что проект компилируется и прошивается на ESP32-C3

2. **Подключение к Wi-Fi**
   - ESP32 подключается к Wi-Fi сети (SSID/пароль захардкожены или в конфиге)
   - Логирование статуса подключения в Serial

3. **Отправка тестовой телеметрии на бэкенд**
   - Формирование JSON-пакета с захардкоженными тестовыми данными (без реальной CAN-шины на этом этапе)
   - Отправка `POST /api/telemetry` на бэкенд Тимура по HTTP
   - Отправка `POST /api/device/ping` heartbeat
   - Заголовок `X-Device-Key` с тестовым ключом
   - Цикл отправки каждые 5 секунд

### Результат

ESP32 включается → подключается к Wi-Fi → каждые 5 секунд отправляет тестовый JSON на бэкенд. В Serial-мониторе видно статус отправки (200 OK или ошибка).

### Пример тестового пакета

```json
{
  "deviceId": "ESP32-TEST-001",
  "timestamp": "2026-03-22T14:30:00Z",
  "data": {
    "rpm": 1500,
    "speed": 60,
    "coolantTemp": 85,
    "oilPressure": 3.2,
    "fuelLevel": 70,
    "voltage": 13.6,
    "dtcCodes": []
  }
}
```

### Зависимость

- Нужен IP/URL бэкенда Тимура и рабочий эндпоинт `POST /api/telemetry`

## Тимур — Бэкенд + Фронтенд

### Задачи (Бэкенд)

1. **Инициализация проекта**
   - Создать ASP.NET Core Web API проект в репозитории (`/backend`)
   - Подключить Entity Framework Core + Npgsql (PostgreSQL)
   - Настроить `appsettings.json` с connection string к PostgreSQL

2. **База данных — начальная схема**
   - Создать модели и миграцию для основных таблиц:

   | Таблица      | Поля (минимум для этапа)                                              |
   |--------------|-----------------------------------------------------------------------|
   | `vehicles`   | Id, Name, LicensePlate                                                |
   | `devices`    | Id, SerialNumber, VehicleId, ApiKey, IsOnline, LastPingAt             |
   | `telemetry`  | Id, DeviceId, Timestamp, Rpm, Speed, CoolantTemp, OilPressure, FuelLevel, Voltage, DtcCodes (JSON) |
   | `users`      | Id, Role (admin/driver), TelegramId, VehicleId                       |
   | `refuels`    | Id, VehicleId, UserId, Timestamp, Liters, Cost                       |
   | `alerts`     | Id, VehicleId, Timestamp, Severity, Description, IsRead              |

   - Применить миграцию, убедиться что таблицы создаются
   - Добавить seed-данные: 1 тестовая фура, 1 устройство с ApiKey, 1 admin, 1 driver

3. **Эндпоинты для устройства**
   - `POST /api/telemetry` — принять JSON, проверить `X-Device-Key`, сохранить в БД
   - `POST /api/device/ping` — обновить `IsOnline = true`, `LastPingAt = now`

4. **Эндпоинты для фронтенда (минимум)**
   - `GET /api/vehicles` — список фур
   - `GET /api/vehicles/{id}/telemetry?from=&to=` — телеметрия с фильтрацией
   - `GET /api/vehicles/{id}/alerts` — алерты по фуре
   - `GET /api/vehicles/{id}/refuels` — заправки по фуре

5. **Эндпоинт для Telegram-бота**
   - `POST /api/telegram/webhook` — принимает Update от Telegram, парсит команду, возвращает ответ
   - Обработка `/start` — привязка Telegram ID к водителю
   - Обработка `/refuel <литры> <руб>` — сохранение заправки в `refuels`
   - Обработка `/status` — возврат последней телеметрии фуры водителя
   - Отправка ответа через Telegram Bot API `sendMessage`

### Задачи (Фронтенд)

6. **Инициализация проекта**
   - Создать Vue 3 + Vite проект (`/frontend`)
   - Установить минимум зависимостей (axios или fetch для API, vue-router)

7. **Страница «Список фур»**
   - Запрос `GET /api/vehicles`
   - Таблица: название, номер, статус устройства (онлайн/оффлайн)

8. **Страница «Детали фуры»**
   - Запрос `GET /api/vehicles/{id}/telemetry`
   - Таблица или простой список с последними данными телеметрии (без графиков — просто числа)
   - Блок с последними алертами
   - Блок с заправками

### Результат

- Бэкенд запускается, БД создаётся с тестовыми данными
- ESP32 Сергея отправляет данные → они появляются в БД
- Фронтенд показывает список фур и данные телеметрии по каждой
- Webhook принимает команды от Telegram-бота Пайрава

## Пайрав — Telegram-бот

### Задачи

1. **Создать бота**
   - Зарегистрировать бота через @BotFather, получить Bot Token
   - Создать проект бота в репозитории (`/telegram-bot`)
   - Выбрать язык/фреймворк (Python + python-telegram-bot, Node.js + telegraf, или C# — на усмотрение)

2. **Настроить webhook**
   - Зарегистрировать webhook URL бота на эндпоинт Тимура: `POST /api/telegram/webhook`
   - Либо: если бот работает отдельно — поллинг (long polling), а бот сам ходит в API Тимура

3. **Реализовать команды**

   | Команда                 | Что делает                                                    |
   |-------------------------|---------------------------------------------------------------|
   | `/start`                | Отправляет приветствие, регистрирует Telegram ID на бэкенде   |
   | `/refuel <литры> <руб>` | Отправляет данные заправки на бэкенд, показывает подтверждение |
   | `/status`               | Запрашивает последние параметры фуры, показывает водителю     |
   | `/alerts`               | Запрашивает последние алерты, показывает водителю             |

4. **Формат сообщений**
   - Ответы бота в читаемом виде, например:

   ```
   🚛 Статус фуры А123БВ:
   Скорость: 75 км/ч
   Обороты: 1800
   Температура ОЖ: 88°C
   Давление масла: 3.4 бар
   Топливо: 55%
   ```

### Результат

Водитель пишет боту `/status` → бот обращается к API Тимура → получает данные → отправляет водителю красивое сообщение. Команда `/refuel 80 5600` сохраняет заправку в БД и подтверждает водителю.

### Зависимость

- Нужен рабочий бэкенд Тимура с эндпоинтами `/api/telegram/webhook` или REST API для бота

## Порядок работы

Участники могут работать параллельно, но есть точки синхронизации:

```
Неделя 1:
├── Сергей: настройка среды, Wi-Fi, отправка тестового JSON (пока на httpbin или локальный сервер)
├── Тимур:  инициализация .NET + PostgreSQL, схема БД, эндпоинты для устройства
└── Пайрав: регистрация бота, каркас проекта, реализация команд (пока с моковыми данными)

Точка синхронизации 1 — бэкенд готов принимать телеметрию:
├── Сергей: переключает ESP32 на бэкенд Тимура, проверяет что данные доходят
└── Тимур:  проверяет что данные от ESP32 сохраняются в PostgreSQL

Точка синхронизации 2 — бэкенд отдаёт данные:
├── Тимур:  эндпоинты для фронтенда и бота готовы
├── Пайрав: переключает бота на реальный API Тимура, проверяет команды
└── Тимур:  запускает фронтенд, проверяет отображение данных

Финал этапа — сквозной тест:
  ESP32 отправляет данные → данные в БД → видны на фронтенде
  Водитель вносит заправку → заправка в БД → видна на фронтенде
  /status в боте → данные от ESP32 приходят водителю
```

---

## Чего НЕ делаем на этом этапе

- CAN-шина (реальное подключение к фуре) — только тестовые данные
- AI-анализ и нейросеть
- WebSocket (real-time обновления на фронте)
- JWT-авторизация и разграничение прав
- Графики и аналитика
- Буферизация данных на ESP32
- Красивый UI
- Деплой на сервер
